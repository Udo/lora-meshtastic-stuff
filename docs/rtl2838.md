# RTL2838 SDR Workflow

This repo includes a project-local setup wrapper for the USB RTL2832/RTL2838 SDR dongle detected on this machine as `0bda:2838 Realtek RTL2838 DVB-T`.

The wrapper lives at `setup/rtl2838.sh` and uses this layout:

- `rtl2838/vendor/rtl-sdr/`: pinned upstream source checkout
- `rtl2838/vendor/rtl_433/`: pinned upstream source checkout
- `rtl2838/vendor/acarsdec/`: pinned upstream source checkout
- `rtl2838/build/rtl-sdr/`: local build tree
- `rtl2838/build/rtl_433/`: local build tree
- `rtl2838/build/acarsdec/`: local build tree
- `rtl2838/local/bin/`: built utilities such as `rtl_test`, `rtl_sdr`, `rtl_fm`, `rtl_433`, and `acarsdec`
- `rtl2838/captures/`: raw IQ and WAV outputs
- `rtl2838/reports/`: rendered waterfall/PSD PNGs and markdown summaries

## Typical Flow

Inspect the current state:

```bash
./setup/rtl2838.sh info
./setup/rtl2838.sh doctor
```

Build repo-local tools:

```bash
./setup/rtl2838.sh bootstrap
```

Probe the kernel SDR interface that Ubuntu already exposes as `/dev/swradio0`:

```bash
./setup/rtl2838.sh probe-kernel
```

Run the libusb-based `rtl_test` utility after bootstrap:

```bash
./setup/rtl2838.sh rtl-test
```

Capture raw CU8 IQ through the kernel V4L2 SDR path:

```bash
./setup/rtl2838.sh capture-v4l2 868000000 10
```

Render a PSD plus waterfall for a raw capture:

```bash
./setup/rtl2838.sh analyze rtl2838/captures/some-file.cu8 868000000
```

Run the EU868 monitoring demo:

```bash
./setup/rtl2838.sh eu868-demo 10
```

## Decoder Monitors

The repo now includes a small family of terminal decoders with a shared UI and optional JSONL event logging via `--log-file`.

- `adsb-monitor`: ADS-B / Mode S at `1090 MHz` via `readsb`.
- `rtl433-monitor`: consumer ISM sensors via repo-local `rtl_433`. Presets: `433`, `868`, `915`; default is `868`.
- `aprs-monitor`: APRS on `144.800 MHz` in Europe using `rtl_fm` piped into `multimon-ng`.
- `ais-monitor`: AIS marine traffic via `rtl_ais` on `161.975 MHz` and `162.025 MHz`.
- `weather-alert-monitor`: SAME/EAS alert monitoring via `rtl_fm` and `multimon-ng`, defaulting to `162.550 MHz`.
- `acars-monitor`: ACARS aircraft text messages via `acarsdec`, defaulting to `131.525`, `131.725`, and `131.825 MHz`.
- `rds-monitor`: FM RDS / RBDS via `rtl_fm` piped into repo-local `redsea`.
- `vdl2-monitor`: VDL2 aviation data via repo-local `vdlm2dec`.
- `pager-monitor`: POCSAG pager traffic via `rtl_fm` and `multimon-ng`.
- `noaa-apt-capture`: NOAA weather satellite APT capture to WAV and PNG via repo-local `aptdec`.
- `lora-monitor`: one-shot LoRa raw hex dump decoding via repo-local `rtl_sdr` capture plus repo-local `gr-lora`.

Example direct launches:

```bash
python3 tools/rf_adsb_monitor.py --log-file rtl2838/logs/adsb.jsonl
python3 tools/rf_rtl433_monitor.py --preset 868 --log-file rtl2838/logs/rtl433.jsonl
python3 tools/rf_aprs_monitor.py --log-file rtl2838/logs/aprs.jsonl

```

## Notes

- The script pins `rtl-sdr` to tag `v2.0.2`, `rtl_433` to tag `25.02`, and `acarsdec` to tag `acarsdec-3.7` by default, and also vendors `redsea`, `vdlm2dec`, `aptdec`, and `gr-lora`.
- Override the pinned ref with `RTL2838_VENDOR_REF=<tag-or-commit>`.
- Override the pinned `rtl_433` ref with `RTL2838_RTL433_VENDOR_REF=<tag-or-commit>`.
- Override the pinned `acarsdec` ref with `RTL2838_ACARSDEC_VENDOR_REF=<tag-or-commit>`.
- Override the kernel SDR device path with `RTL2838_DEVICE=/dev/swradio0`.
- Building `rtl-sdr` on Ubuntu/Debian requires `libusb-1.0-0-dev` in addition to `git`, `cmake`, `make`, `gcc`, and `pkg-config`.
- `bootstrap` now installs the apt-managed host prerequisites for every shipped SDR tool on Ubuntu/Debian and builds `rtl_433`, `acarsdec`, `redsea`, `vdlm2dec`, `aptdec`, and `gr-lora` locally too.
- V4L2 capture writes raw interleaved `CU8` I/Q bytes.
- The EU868 demo is a spectrum monitor and visualizer, not a LoRa packet decoder.
- `eu868-live` and `live-waterfall` use the kernel `/dev/swradio0` path directly and render in the terminal.
- The default live view now uses a combined `eu868-wide` window centered at `868.475 MHz` with `3.2 MS/s`, which spans roughly `866.875 MHz` to `870.075 MHz`.
- The default live rendering mode is `blocks`, which uses Unicode block glyphs instead of ASCII letters.
- Character presets are `blocks`, `braille`, `ascii`, and `dense`. You can also pass your own glyph string in low-to-high intensity order.
- The default live normalization mode is `slow`, which keeps a slowly adapting floor/ceiling across time so background noise does not look like constant traffic.
- Slow normalization now starts conservatively with extra headroom and then becomes more sensitive over the next few rows instead of starting fully bright.
- The default rendered-row averaging is `3` FFT frames, which increases visible update rate compared with the earlier slower view.
- The direct live viewer now uses `--profile` as the full band selection, with `--preset` retained as an alias for the same thing.
- `--markers` now adds extra marker frequencies as a comma-separated list such as `144.390,iss=145.825`; it no longer selects a band profile.
- Named live presets now include consumer listening bands such as `am-broadcast`, `shortwave-49m`, `shortwave-31m`, `weather`, `marine-vhf`, `airband`, and `fm-broadcast`.
- Hobbyist and utility presets now include `cb-27mhz`, `ham-10m`, `ham-6m`, `ham-2m`, `ham-70cm`, `murs`, `ism433`, `pmr446`, `frs-gmrs`, `ism915`, `adsb1090`, plus the EU868 variants.
- The live waterfall now also exposes presets aligned with the decoder tools: `adsb-monitor`, `aprs-monitor`, `ais-monitor`, `acars-monitor`, `rds-monitor`, `vdl2-monitor`, `pager-monitor`, `noaa-apt-monitor`, `lora-monitor`, `weather-alert-monitor`, `rtl433-433`, `rtl433-868`, and `rtl433-915`.
- The decoder-aligned waterfall presets now use richer marker sets too: `aprs-monitor` shows common APRS/packet references, `ais-monitor` shows AIS A/B plus marine channel 16, `acars-monitor` shows the common VHF ACARS channels, and the `rtl433-*` presets show representative ISM reference frequencies instead of only one line.
- `max-span` is the explicit widest-instantaneous-bandwidth preset for this RTL2838 workflow. It uses the current practical max span of about `3.2 MHz`.
- A true one-shot live sweep across the receiver's full advertised `25 MHz` to `1760 MHz` range is not feasible here; the hardware only observes one slice at a time, so `max-span` is the fallback.
- `broadband-868` is the intentionally broad max-span preset for a wide UHF spectrum look without marker overlays.
- `adsb-monitor` now uses `readsb` as the backend and presents the live `aircraft.json` view in the shared TUI.
- `rtl433-monitor` is the next decoder integration. It wraps a repo-local `rtl_433` build when available, parses JSON output, and shows live consumer sensor / ISM device activity.
- `rtl433-monitor` now has named tuning presets for common sensor bands: `433`, `868`, and `915`.
- The default `rtl433-monitor` preset is `868`, which is the most sensible default for common EU sensor traffic.
- `aprs-monitor`, `weather-alert-monitor`, and `ais-monitor` use libusb mode and will switch the dongle automatically when needed.
- `aprs-monitor`, `weather-alert-monitor`, `ais-monitor`, and `fm` now auto-install their apt-managed host dependencies when missing, prompting for `sudo` when required.
- `acars-monitor` also uses libusb mode and now prefers the repo-local vendored `acarsdec` build from `bootstrap`.
- `rds-monitor`, `vdl2-monitor`, and `pager-monitor` also use libusb mode and follow the same shared TUI/logging pattern.
- `noaa-apt-capture` and `lora-monitor` are capture-first workflows. They still switch to libusb mode automatically, but they are not fullscreen TUIs.
- All decoder monitors accept `--log-file` and append newline-delimited JSON events without disabling the live TUI.
- `row` normalization is still available if you want maximum contrast on every individual row, but it will make quiet bands look busier.
- If a libusb tool fails to claim the device, first stop any other SDR software that may have opened it.
